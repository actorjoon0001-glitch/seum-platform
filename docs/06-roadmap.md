# 06. 개발 우선순위 / 로드맵

## 1. 우선순위 결정 기준

1. **돈을 만드는 경로 먼저** — 상담(CRM) → 견적 → 계약이 핵심 가치 흐름.
2. **기반은 한 번에 제대로** — 멀티테넌시/인증/RLS는 모든 기능의 토대라 0순위.
3. **무거운 기술(3D)은 가치 검증 후** — CRM/견적으로 PMF 확인 뒤 3D 투자.
4. **AI/전자서명/PG는 확장 슬롯** — 구조만 미리 열어두고 나중에.

## 2. 마일스톤

### 🟥 M0 — 기반 (Foundation) · 필수 선행
> 이게 없으면 어떤 기능도 안전하게 못 만든다.

- [ ] Next.js 15 + TS + Tailwind + ESLint/Prettier 셋업
- [ ] Supabase 프로젝트, 환경변수(`config/env.ts` zod 검증)
- [ ] 인증: 로그인/세션/`middleware.ts` 가드
- [ ] 멀티테넌시: `organizations/members/profiles` + 헬퍼함수 + RLS
- [ ] 활성 Org 스위처, TanStack Query / Zustand 셋업
- [ ] DB 타입 자동생성 파이프라인, 마이그레이션 워크플로우
- [ ] 공유 UI 키트(Button/Input/Table/Modal), 앱 레이아웃

**완료 기준(DoD):** 두 개의 테스트 org가 서로 데이터를 절대 볼 수 없다(RLS 검증).

### 🟧 M1 — CRM (고객관리) · 첫 가치
- [ ] 고객 등록(이름/연락처/주소/토지/희망평수/예산/유입경로)
- [ ] 고객 목록 + 검색(이름/연락처) + 필터(상태/담당)
- [ ] 고객 상세 + 메모(`customer_notes`)
- [ ] 상태 파이프라인 UI(칸반 또는 상태배지) + 전이
- [ ] RBAC: 영업/관리자 권한 반영

**DoD:** 영업이 리드를 등록→상담중→견적발송까지 상태로 관리 가능.

### 🟨 M2 — 카탈로그 + 실시간 견적 · 매출 직결
- [ ] 모델 CRUD(평형/가격/이미지/설명/옵션)
- [ ] 옵션 관리(포치/데크/다락/태양광/창호/황토 …)
- [ ] 견적 빌더: 모델+옵션 선택 → **즉시 가격 반영**(클라 미리보기)
- [ ] 서버 가격 확정 계산(`/api/quotes`) + `quote_items` 스냅샷
- [ ] **견적서 PDF 생성** + Storage 저장 + 발송(상태 sent)
- [ ] 견적→고객 상태 연동(견적발송/계약대기)

**DoD:** 옵션 클릭 즉시 총액이 바뀌고, 신뢰 가능한 PDF 견적서가 나온다.

### 🟩 M3 — 계약 + 시공관리 · 운영 마무리
- [ ] 견적→계약 전환, 계약금/중도금/잔금 생성
- [ ] 계약서 PDF, 납입 상태 추적(정산 역할)
- [ ] 계약완료 시 `projects` 자동 생성
- [ ] 시공 단계(계약→설계→허가→생산→운송→설치→완료) 전이
- [ ] 시공 사진 업로드(서명URL) + 진행 피드 + 일정

**DoD:** 계약부터 시공 완료까지 한 화면에서 추적된다.

### 🟦 M4 — 대시보드 · 가시성
- [ ] 매출/계약건수/고객수/전환율 집계 API
- [ ] 월별 통계 차트, 파이프라인 퍼널
- [ ] Realtime 실시간 반영
- [ ] 역할별 대시보드(영업=본인실적, 정산=매출)

**DoD:** 대표가 로그인 즉시 핵심 지표를 본다.

### 🟪 M5 — 3D 설계 시스템 · 차별화
> M1~M4로 가치 검증 후 투자. **Scene Spec** 구조는 M0~M2에서 미리 정의.

- [ ] Scene Spec 스키마/검증/마이그레이션(`features/designer/spec`)
- [ ] R3F 캔버스: 3D 보기 + 평면도(2D) 토글
- [ ] 가구 배치 / 창문·문 위치 변경(드래그)
- [ ] `scenes` 저장/불러오기, 견적·모델 연동
- [ ] 썸네일 렌더 저장

**DoD:** 설계자가 모델 기반으로 창/문/가구를 배치하고 저장한다.

### ⬛ M6 — 확장 슬롯 (Future)
- [ ] AI 자동설계(요구사항 → Scene Spec 생성)
- [ ] 전자서명 연동(`contracts.signed_at` 활용)
- [ ] 결제(PG) 연동, 알림(카톡/이메일)
- [ ] 업체별 권한 커스터마이즈(`permissions jsonb`)

   │                        │

## 4. 횡단 관심사 (모든 마일스톤 공통)
- 각 기능마다: zod 검증, RBAC 가드, RLS 정책, 빈/로딩/에러 상태.
- 마이그레이션은 항상 PR에 포함, 롤백 고려.
- 핵심 로직(pricing, 상태 전이, 권한)은 단위 테스트 필수.

## 5. 권장 진행 방식
- 마일스톤 = 1~2주 스프린트 단위로 분할.
- 각 마일스톤 종료 시 데모 + DoD 체크.
- M0는 절대 건너뛰지 않는다(멀티테넌시 보안의 토대).
